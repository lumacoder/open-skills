# open-skills V5 架构演进建议

> 基于 HyperFrames skill 封装策略的深度研究，提出 open-skills 下一代架构优化方向。
>
> **研究对象**: [heygen-com/hyperframes](https://github.com/heygen-com/hyperframes) —— 一个生产级的多层级 skill 体系 + block registry 项目。

---

## 一、HyperFrames 核心策略提炼

### 1.1 多层级 Skill 拆分（Multi-Tier Skill Architecture）

HyperFrames 不是用一个大而全的 SKILL.md 覆盖所有场景，而是按**职责边界**拆分为多个独立的 skill：

| Skill | 职责 | 触发条件 |
|-------|------|---------|
| `hyperframes` | 核心框架用法：composition 结构、数据属性、媒体处理 | 用户要写 HTML composition |
| `hyperframes-cli` | CLI 命令：init、lint、preview、render、doctor | 用户提到命令行操作 |
| `hyperframes-registry` | Block/Component 的发现与安装 | 用户提到 `hyperframes add` 或 block |
| `gsap` | GSAP API、timelines、easing、ScrollTrigger | 用户要做动画 |
| `website-to-hyperframes` | 将网站捕获为视频的完整工作流 | 用户粘贴 URL |

**核心洞察**：每个 skill 只负责一个**内聚的领域**，通过 trigger 条件让 LLM 自主选择调用哪个。

### 1.2 Progressive Disclosure（渐进式披露）

文档结构不是平铺的，而是分层级的：

```
表层：触发条件 + 核心规则（LLM 首次加载时读取）
  ↓
中层：工作流步骤 + 命令速查（使用时按需读取）
  ↓
深层：API 细节 + 实现参考（通过 cross-reference 跳转）
```

HyperFrames 主 skill upfront 只放 "What/Structure/Timing/Layout/Animate" 五步法，具体 transitions 的 20+ 种实现细节全部 deferred 到 `/references/`。

### 1.3 Hard Gates（硬门控）

用 `<HARD-GATE>` 标签明确阻塞前置条件：

```markdown
<HARD-GATE>
- 必须已运行 `npx hyperframes init`
- 必须存在 `compositions/index.html`
- 必须已配置 `data-composition-id`
</HARD-GATE>
```

LLM 在生成代码前必须先检查这些条件，不满足则引导用户完成前置步骤。

### 1.4 编号规则系统（Numbered Constraints）

约束不是散落的全大写句子，而是**编号化的 Never do 列表**：

```markdown
## 绝对禁止

1. 永远不要直接操作 DOM，始终使用 GSAP 的 `targets` 参数
2. 永远不要在 `useGSAP()` 外部调用 `gsap.to()` 等动画方法
3. 永远不要对同一元素同时使用 CSS animation 和 GSAP tween
...
11. 永远不要省略 `data-composition-id`，否则 preview 无法工作
```

编号的好处：
- **引用精确**：review 时可以说 "违反了规则 #7"
- **LLM 记忆锚点**：编号比长句子更容易被模型记住
- **渐进添加**：新增规则只需递增编号，不破坏已有结构

### 1.5 WRONG / RIGHT 对比示例

不是只告诉 LLM "应该怎么做"，而是**成对展示**违规与合规：

```markdown
**WRONG** — 在 `useGSAP()` 外部调用：
```js
gsap.to('.box', { x: 100 }); // ❌ 没有正确的 cleanup
```

**RIGHT** — 在 `useGSAP()` 作用域内调用：
```js
useGSAP(() => {
  gsap.to('.box', { x: 100 }); // ✓ 自动 cleanup
});
```
```

### 1.6 分层 Registry Catalog

Registry 不是单层清单，而是**两级结构**：

```
registry.json          ← 轻量目录（50 items × 2 fields = 100 行）
  └── items/{name}/
        └── registry-item.json   ← 完整元数据（每个 item 独立文件）
```

**registry.json**（目录级）：
```json
{
  "$schema": "https://hyperframes.heygen.com/schema/registry.json",
  "name": "hyperframes",
  "homepage": "https://hyperframes.heygen.com",
  "items": [
    { "name": "data-chart", "type": "hyperframes:block" },
    { "name": "grain-overlay", "type": "hyperframes:component" }
  ]
}
```

**registry-item.json**（详情级）：
```json
{
  "name": "data-chart",
  "type": "hyperframes:block",
  "title": "Animated Data Chart",
  "description": "...",
  "tags": ["chart", "data", "animation"],
  "dimensions": { "width": 1920, "height": 1080 },
  "duration": 15,
  "files": ["index.html", "styles.css"]
}
```

**优势**：
- 目录级文件极小，CLI 可以快速拉取做本地缓存
- 详情按需加载，不阻塞发现流程
- 新增 item 只需在 `items/` 下新建目录，不需要修改主清单

### 1.7 Type 系统与 Schema 约束

每个 item 都有明确的 `type`，且 registry 顶部声明 `$schema`：

| Type | 语义 |
|------|------|
| `hyperframes:example` | 完整示例项目，可 `init` |
| `hyperframes:block` | 独立 composition，可 `add` 并 wiring |
| `hyperframes:component` | 代码片段，需手动粘贴到 host |

Schema 让 registry 具备**自校验能力**，CI 可以验证新增 item 的字段完整性。

### 1.8 CLI-first, Non-interactive by Default

HyperFrames CLI 的设计理念："**The CLI is non-interactive by default, designed for agent-driven workflows**"

```bash
# 全部支持 --json 输出，方便 agent 解析
npx hyperframes add data-chart --json
npx hyperframes add data-chart --no-clipboard   # CI/headless

# 安装即输出 wiring snippet
npx hyperframes add data-chart
# → 输出文件路径 + HTML snippet 到剪贴板
```

### 1.9 参数表格化 + 使用建议

复杂命令的参数不用长段落描述，而是用**4列表格**，且 notes 列提供**使用建议**而非纯技术规格：

| Flag | Options | Default | Notes |
|------|---------|---------|-------|
| `--quality` | draft, standard, high | standard | draft for iterating, high for final delivery |
| `--fps` | 30, 60 | 30 | 60fps doubles render time |

### 1.10 安装即 Wiring（Install-to-Wire）

Block 安装后不只是写入文件，CLI 还会输出**可直接粘贴的 wiring snippet**：

```bash
$ npx hyperframes add data-chart
✓ Installed to: compositions/data-chart.html

Paste this into your host composition:

  <div
    data-composition-id="data-chart"
    data-composition-src="compositions/data-chart.html"
    data-start="2"
    data-duration="15"
    data-track-index="1"
  ></div>
```

这大大降低了用户（尤其是 agent）将 block 集成到现有项目的门槛。

## 二、open-skills 现状与差距分析

### 2.1 当前架构（V4）

| 维度 | 现状 |
|------|------|
| Skill 结构 | 单一 SKILL.md 对应一个 skill，无职责拆分 |
| Registry | 单层 `registry/skills.json`，所有元数据集中 |
| Schema | 无 `$schema` 声明 |
| Type 系统 | 无，所有 skill 同质化处理 |
| 约束表达 | 依赖 skill 作者自行组织，无标准格式 |
| CLI 设计 | 交互式为主（TUI 菜单），agent 场景支持弱 |
| 安装后 | 静态文件写入，无 wiring/集成指引 |
| 参数文档 | 无统一规范 |
| Quality checks | 无内置 lint/验证机制 |

### 2.2 与 HyperFrames 的关键差距

| 差距 | 影响 |
|------|------|
| **无 Type 系统** | 无法区分 "可独立使用的 skill" vs "需要 wiring 的 snippet" vs "完整项目模板" |
| **单层 Registry** | `skills.json` 膨胀后拉取变慢，新增 skill 需修改中心文件 |
| **无 Hard Gates** | LLM 可能在条件不满足时直接生成代码，导致错误 |
| **无编号规则** | 约束散落在文档各处，难以 review 和引用 |
| **CLI 交互式为主** | Agent 无法直接调用 `open-skills --editor cursor --category frontend --skill react` |
| **安装即结束** | 用户/agent 不知道安装后如何激活或使用 skill |
| **无 Progressive Disclosure** | 所有内容平铺，LLM context 被不相关的深细节占满 |

## 三、V5 架构演进建议

### 建议 1：引入 Skill Type 系统

为 skill 增加 `type` 字段，区分不同使用模式：

```json
{
  "name": "react-best-practices",
  "type": "skill:standalone",     // 独立 skill，安装即生效
  ...
}
```

| Type | 说明 | 安装行为 |
|------|------|---------|
| `skill:standalone` | 独立 skill，有完整 SKILL.md | 直接写入目标目录 |
| `skill:snippet` | 代码片段/规则集合，需集成到宿主 | 写入后输出 wiring 指引 |
| `skill:template` | 项目模板/脚手架 | `init` 行为，生成完整项目 |
| `skill:workflow` | 多步骤工作流指引 | 写入后提示按步骤执行 |

**价值**：
- CLI 可以根据 type 自动调整安装后的行为
- TUI 可以在选择时给出不同的安装提示
- 为后续 `open-skills init <template>` 命令铺路

### 建议 2：分层 Registry 结构

将当前的 `registry/skills.json` 拆分为两级：

```
registry/
├── index.json              ← 轻量目录（~100 行）
│   {
│     "$schema": "./schema.json",
│     "name": "open-skills",
│     "version": "5.0.0",
│     "homepage": "https://open-skills.dev",
│     "items": [
│       { "name": "react-best-practices", "type": "skill:standalone" },
│       { "name": "frontend-design", "type": "skill:workflow" }
│     ]
│   }
└── items/
    ├── react-best-practices/
    │   └── registry-item.json   ← 完整元数据
    ├── frontend-design/
    │   └── registry-item.json
    └── ...
```

**registry-item.json 字段扩展**：

```json
{
  "name": "react-best-practices",
  "type": "skill:standalone",
  "displayName": "React Best Practices",
  "description": "React & Next.js 性能优化与工程实践指南",
  "category": "frontend",
  "tags": ["react", "nextjs", "performance"],
  "origin": {
    "type": "github",
    "ref": "vercel-labs/agent-skills",
    "path": "skills/react-best-practices"
  },
  "author": "Vercel Labs",
  "version": "2.1.0",
  "license": "MIT",
  "wiring": {
    "requires": ["node", "react"],
    "postInstall": "SKILL.md 已安装到 .claude/skills/，无需额外配置"
  }
}
```

**价值**：
- CLI 启动时只拉取 `index.json`（~2KB），启动速度极快
- 详情按需加载，浏览分类时只拉取该分类下的 item
- 新增 skill 无需修改中心文件，PR 冲突减少

### 建议 3：CLI 非交互模式强化

当前 CLI 默认进入交互式 TUI，对 agent 不友好。建议：

```bash
# 当前：只能交互式选择
open-skills

# V5 建议：支持全参数化调用
open-skills install \
  --editor claude-code \
  --scope global \
  --skills react-best-practices,typescript-patterns

# 输出机器可读格式
open-skills install --skills react-best-practices --json
# → {"success": true, "installed": [...], "wiring": [...]}

# 非交互模式下自动确认
open-skills install --skills react-best-practices --yes

# 检查前置条件
open-skills install --skills react-best-practices --dry-run
```

**TUI 保持**，但增加 `--simple` 降级模式和全参数化支持。

### 建议 4：安装后 Wiring 指引

安装完成后，根据 skill type 输出 wiring 信息：

```bash
$ open-skills install --skills shadcn-ui

✓ Installed shadcn-ui to ~/.claude/skills/shadcn-ui/

📋 Wiring Guide:
  1. SKILL.md 已就绪，Claude 会自动加载
  2. 如需 Tailwind 配置，参考 SKILL.md 中 "Tailwind Setup" 章节
  3. 常用命令速查：
     - `npx shadcn add <component>`
     - `npx shadcn init`
```

对于 `skill:snippet` 类型：

```bash
$ open-skills install --skills react-hooks-snippets

✓ Installed react-hooks-snippets to ~/.claude/skills/react-hooks-snippets/

⚠ This is a snippet skill. Paste these rules into your existing files:

  - useEffect patterns → paste into your component files
  - custom hook templates → paste into your hooks/ directory
```

### 建议 5：引入 Skill 内容规范（Content Spec）

为 skill 作者提供**内容格式规范**，提升 LLM 消费质量：

```markdown
---
name: my-skill
description: |
  Use when: 用户需要...
  Use after: 必须先完成...
version: 1.0.0
---

# my-skill

## 🚧 Hard Gates

1. 必须已安装 Node.js >= 18
2. 必须已初始化 git 仓库

## 📋 Core Rules

1. **永远不要** 使用 var，始终使用 const/let
2. **永远不要** 在 React 组件中直接修改 props
3. **总是** 为函数组件添加 displayName

## 🔄 Workflow

### Step 1: Scaffold
```bash
npm create vite@latest
```

### Step 2: Configure
...

## 📚 References

- [Advanced Patterns](references/advanced.md) — read when 用户需要...
- [API Cheat Sheet](references/api.md) — read when 用户需要...
```

**规范要点**：
- `Use when:` / `Use after:` 触发条件前置
- `Hard Gates` 编号化前置条件
- `Core Rules` 编号化约束
- `Workflow` 按步骤组织
- `References` 深细节 deferred

### 建议 6：引入 Quality Checks

在 `open-skills validate` 中增加内容级校验：

```bash
$ open-skills validate

✓ Registry structure valid
✓ All source URLs reachable
⚠ Skill "foo-bar" missing Hard Gates section
⚠ Skill "baz-qux" has unnumbered rules (line 45)
✗ Skill "bad-skill" missing trigger conditions in description
```

校验项：
- `description` 中是否包含 `Use when:` 触发条件
- 是否存在 `Hard Gates` / `Core Rules` 章节
- 规则是否编号化
- 是否包含 `Workflow` 或 `References` 章节
- `SKILL.md` 文件大小是否超过建议阈值（如 500 行则提示拆分）

### 建议 7：open-skills 自身的 Skill 拆分

当前 open-skills 只有一个入口 skill，建议按 HyperFrames 模式拆分：

```
skills/
├── open-skills/           ← 主 skill：介绍 open-skills 是什么
│   └── SKILL.md
├── open-skills-cli/       ← CLI 使用 skill：命令速查、参数说明
│   └── SKILL.md
├── open-skills-registry/  ← Registry 贡献 skill：如何注册新 skill
│   └── SKILL.md
└── open-skills-authoring/ ← Skill 编写 skill：内容规范、最佳实践
    └── SKILL.md
```

这样用户问 "怎么安装 skill" 时 LLM 加载 `open-skills-cli`，问 "怎么写一个好的 skill" 时加载 `open-skills-authoring`，context 更聚焦。

## 四、优先级与实施路径

### Phase 1：基础设施（2 周）

| 任务 | 说明 |
|------|------|

## 五、关键决策记录

### 5.1 为什么参考 HyperFrames 而非其他项目？

HyperFrames 与 open-skills 高度对标：
- 都是 CLI-first 的 skill/block 分发工具
- 都面临 "skill 怎么写才能让 LLM 更好地消费" 的问题
- 都有 registry + 安装引擎的架构
- HyperFrames 已经在生产环境运行，验证了其策略的有效性

### 5.2 为什么保留 TUI 同时强化非交互模式？

TUI 对新用户友好（发现式浏览），非交互模式对 agent 友好（确定性调用）。两者不冲突，是不同场景的互补。

### 5.3 为什么引入 Type 系统而不是扩展 tags？

Tags 是描述性的（"这是什么"），Type 是指令性的（"怎么用它"）。安装引擎需要根据 type 决定安装后的行为，这不是 tags 能承载的语义。

### 5.4 为什么建议 open-skills 自身也拆分 skill？

open-skills 作为 "skill 的安装工具"，自身也应该遵循它倡导的 skill 最佳实践。拆分后：
- 每个 skill 的 context 更小、更聚焦
- LLM 加载速度更快
- 用户问题与 skill 的匹配更精确

## 附录：HyperFrames Registry 结构速查

```
hyperframes/
├── skills/                          # AI Agent Skills
│   ├── hyperframes/SKILL.md         # 核心框架 skill
│   ├── hyperframes-cli/SKILL.md     # CLI 命令 skill
│   ├── hyperframes-registry/SKILL.md # Registry 使用 skill
│   ├── gsap/SKILL.md                # GSAP API skill
│   └── website-to-hyperframes/SKILL.md # 工作流 skill
│
├── registry/                        # Block/Component Catalog
│   ├── registry.json                # 轻量目录（$schema + items 列表）
│   └── items/
│       ├── data-chart/
│       │   └── registry-item.json   # 完整元数据
│       ├── flash-through-white/
│       │   └── registry-item.json
│       └── grain-overlay/
│           └── registry-item.json
│
├── packages/                        # 核心框架（npm 包）
│   ├── cli/                         # CLI 工具
│   ├── core/                        # Types, parsers, runtime
│   ├── engine/                      # Puppeteer + FFmpeg
│   └── ...
│
└── .claude/
    └── CLAUDE.md                    # 项目级 AI 配置
```

**关键数字**：
- Registry 目录级文件：~100 行 JSON（50 items × 2 fields）
- Registry 详情级文件：每个 item ~20-50 行 JSON
- Skills：5 个独立 skill，每个 100-300 行 Markdown
- 总 skill 内容量：~1000 行，vs 一个大而全的 1000 行 SKILL.md，后者 LLM 每次都要全量加载
